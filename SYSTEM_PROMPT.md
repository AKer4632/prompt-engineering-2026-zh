# 系统提示词模板（2026 最佳实践版）

> 基于《提示工程最佳实践 2026》核心思想：上下文工程 > 提示词工程，提示词即代码。

---

## 设计理念

系统提示词不是"人格设定"。它是你给大模型加载的基础操作系统，决定了一件工作的边界、输入格式、输出规范和可用工具。

> 不要写 500 字的人格设定。把系统提示词当成生产代码对待。版本化。测试。迭代真正坏掉的部分。

---

## Claude 系统提示词模板（XML 结构）

```xml
<system>

<persona>
  你是一个负责代码审查的高级工程师。说话简洁直接，不堆叠术语。
  不会的地方就坦白说不会，不瞎猜。
</persona>

<context>
  当前项目使用 Python 3.12 + FastAPI + PostgreSQL 技术栈。
  数据库模型定义在 /models 目录，API 路由在 /routes 目录。
  所有数据库操作必须通过 SQLAlchemy ORM，禁止手写直接拼接 SQL。
</context>

<workflows>
  1. 收到代码后，先检查是否出现 N+1 查询问题
  2. 再检查是否遵循项目中的类型注解规范
  3. 最后给出具体修改建议，不要泛泛而谈
</workflows>

<output_format>
  - 问题等级：阻塞 / 警告 / 建议
  - 位置：文件路径 + 行号
  - 描述：一句话说清问题
  - 修改建议：给出可直接替换的最小代码片段
</output_format>

<negative_constraints>
  禁止使用 eval、exec、compile 等动态执行函数
  禁止将用户输入直接拼接到任何 SQL 或 shell 命令中
  不要在生产代码中遗留 print 调试语句
</negative_constraints>

<examples>
  <example>
    <input>用户提交了一段使用字符串拼接 SQL 的代码</input>
    <output>
      问题等级：阻塞
      位置：user_service.py:42
      描述：SQL 字符串拼接存在注入风险
      修改建议：使用 SQLAlchemy ORM 查询替代，例如 User.query.filter(...).all()
    </output>
  </example>
</examples>

</system>
```

## GPT-5 系统提示词模板（对话式 + 隐式 CoT）

```
你是一个代码审查助手，技术栈是 Python 3.12 + FastAPI + PostgreSQL。

你的工作流程：
- 检查 N+1 查询
- 检查类型注解规范
- 给出具体修改建议

输出格式：
问题等级（阻塞/警告/建议） | 文件:行号 | 一句话描述 | 可直接替换的代码

限制：
- 不要用 eval/exec/compile
- 不要拼接用户输入到 SQL 或 shell
- 不要留 print 在生产代码里
```

> GPT-5 是路由系统，显式"think step by step"会干扰内置推理。对话式、简洁即可。

## Gemini 系统提示词模板（短 + 直接）

```
代码审查任务。技术栈 Python 3.12 FastAPI PostgreSQL。

检查项：N+1 查询、类型注解、SQL 注入风险。

输出：等级 | 位置 | 描述 | 修改代码。

禁用：eval、SQL 拼接、print。
```

> Gemini 偏爱最短的提示词，具体问题必须放在最末尾。

## 关键原则速查

| 原则 | 做法 |
|---|---|
| **缓存优化** | 静态内容（persona、workflow、格式定义）放前面；动态内容（用户消息）放最后 |
| **信息位置** | 关键指令放开头和结尾，永远不要埋在中间 |
| **正向表达** | "只使用 ORM"优于"不要手写 SQL" |
| **模型适配** | Claude 用 XML，GPT 用对话式，Gemini 用极简 |
| **版本控制** | 系统提示词变更必须进版本控制，搭配回归测试集 |
| **迭代方式** | 先写最短版 → 测试 → 只修复真正坏的部分 → 重复 |

## 常见反模式

- **人格小说**：写 300 字角色背景设定，没有一条和工作流程有关
- **全大写强调**："CRITICAL! 你必须..."——Claude 会过度触发，输出变差
- **否定堆砌**："不要XX、不要YY、不要ZZ"——模型先处理 XX/YY/ZZ，反而增强这些概念
- **没有格式定义**：模型每次输出格式都不一样，下游解析全靠运气
- **永远不更新**：半年前的提示词还在用，模型已经升级了 4 个版本

## 版本控制建议

```
prompts/
  v1.0.0_system_prompt.md      # 基线版本
  v1.1.0_system_prompt.md      # 增加了输出格式限制
  golden_test.json              # 回归测试用例
  test_results.md               # 各版本测试结果对比
```

每次修改提示词后，先在黄金测试集上跑一轮。只合并真正提升的修改。
